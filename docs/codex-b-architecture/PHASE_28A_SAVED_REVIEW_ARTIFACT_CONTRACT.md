# Phase 28A Saved Review Artifact Contract

Status: proposed
Owner: Codex B
Related plan: `docs/shared/implementation_plan.md` Phase 28A

## Purpose

Phase 27C made review scope visible across Trade Review, Reports, and the
read-only Agent Console. Phase 28A turns a completed review into a durable saved
artifact so users can revisit what was reviewed, when it was reviewed, and which
scope/caveats were in force at generation time.

The product goal is to move Portfolio Copilot from a transient pre-trade checker
to a review workspace with a trustworthy decision journal.

## Core Rule

A saved review artifact is immutable historical context. It must never be
reinterpreted through the current account selector, current Account Details
snapshot, current portfolio context, current market data, current Agent Team
state, or route state.

If the saved artifact lacks a field, the UI must show honest unavailable copy.
It must not reconstruct missing report scope, deterministic summary, or agent
output from newer app state.

## What Phase 28A Should Save

The first durable artifact should preserve only reviewed, display-safe data:

- report title, type, status, and generated timestamps;
- immutable `ReportScopeMetadataRead`;
- deterministic review summary categories and caveats;
- broker/market freshness state at generation time;
- actionability/review mode at generation time;
- optional read-only Agent Team run summary and role outputs, if generated by
  the reviewed Agent Console path;
- source/version labels for the app-owned review pipeline.

The artifact may contain backend-owned display labels and sanitized categories.
It must not contain raw broker/provider/account identifiers, raw account
numbers, raw provider payloads, prompts, LLM traces, raw holdings, raw
positions, tax lots, transaction/order data, raw balances, buying power,
private thresholds, secrets, or API keys.

## Contract Shape

Codex C should define the exact backend schema, but the accepted shape should be
conceptually:

- `SavedReviewArtifactCreateRequest`
  - source run/reference fields that are app-owned and opaque;
  - report title/type;
  - explicit user id route ownership, not a frontend-supplied provider id;
  - no raw account/provider identifiers.
- `SavedReviewArtifactRead`
  - report thread/read metadata;
  - `scope_metadata: ReportScopeMetadataRead | null`;
  - `deterministic_summary: SavedDeterministicReviewSummaryRead | null`;
  - `agent_summary: SavedAgentTeamSummaryRead | null`;
  - `limitations: tuple[str, ...]`;
  - `caveat_codes: tuple[str, ...]`;
  - generated/saved timestamps.
- `SavedDeterministicReviewSummaryRead`
  - reviewed flow and symbol labels when already display-safe;
  - backend-owned review mode/actionability labels;
  - freshness labels;
  - deterministic caveat codes;
  - no raw positions, quantities, balances, lots, or account refs.
- `SavedAgentTeamSummaryRead`
  - run status;
  - role display names and sanitized role summaries if available;
  - provider-neutral warning categories;
  - no prompts, traces, raw provider output, or private evidence rows.

If implementation can reuse existing `ReportThreadRead` /
`ReportThreadDetailRead`, that is preferred over creating parallel report
models, but saved scope and generated-time review data must be persisted rather
than reconstructed.

## Endpoint Direction

Preferred backend direction:

- `POST /users/{uid}/reports/from-trade-review`
  - creates a saved report artifact from a completed reviewed trade-review
    workspace/run reference or from a reviewed sanitized payload.
- `POST /users/{uid}/reports/from-agent-team-run`
  - optional if Agent Team save is separate from deterministic Trade Review.
- Existing:
  - `GET /users/{uid}/reports`
  - `GET /users/{uid}/reports/{thread_id}`

Codex C may propose a narrower endpoint if it better matches the existing
report storage model. Any endpoint must resolve ownership server-side and fail
closed for unknown/cross-user references.

## Frontend Behavior

Frontend work should be small and explicit:

- Add a "Save report" / "Save review snapshot" action only after the backend
  contract passes review.
- The action must save the current reviewed result, not current mutable account
  state.
- Saved Reports must continue to render only saved report data.
- Do not add report editing, report regeneration, trading actions, broker
  actions, or automatic sharing.
- If save fails, show sanitized retryable copy only.

The frontend must not compute report metrics, infer scope, or send raw
broker/account/position data to the report endpoint.

## Agent Team Boundary

Agent Team output may be saved only as sanitized display output from the reviewed
read-only console path.

Do not save:

- prompt text;
- raw LLM request/response bodies;
- provider trace ids;
- tool raw payloads;
- private evidence rows;
- account refs/labels unless already approved for display in the report
  contract;
- cash/position/tax-lot details.

If future work needs prompt/eval trace persistence, it requires a separate ADR
or architecture contract.

## Safety Language

Reports may say:

- "Saved review snapshot"
- "Scope used"
- "Generated from reviewed data available at the time"
- "Manual decision support"

Reports must not say:

- advice, recommendation, guaranteed return;
- safe to trade, ready to trade;
- buy/sell instruction;
- order, execute, place trade, submit trade;
- AI-picked or automated decision wording.

Negated safety copy such as "Not an order recommendation" remains acceptable
when concise and not repeated everywhere.

## Tasks

### P28A-T1 Backend Saved Artifact Contract

Owner recommendation: Codex C

Define the backend schema and endpoint for creating saved review artifacts from
completed Trade Review / Agent Team outputs.

Acceptance:

- Saved artifact persists immutable generated-time scope metadata.
- No reconstruction from current Account Details or selector state.
- No raw identifiers, raw provider payloads, prompts, traces, raw holdings,
  positions, quantities, balances, tax lots, transactions, or order data.
- Unknown/cross-user references fail closed.
- Existing Reports list/detail contract remains backward compatible.

### P28A-T2 Backend Persistence And Projection

Owner recommendation: Codex C

Implement report persistence/projection after P28A-T1 is accepted.

Acceptance:

- Newly saved reports return non-null `scope_metadata` when generated from a
  scoped review.
- Legacy/unknown reports can still return `scope_metadata: null`.
- Tests cover immutable scope, stale/current-state non-reinterpretation, and
  forbidden-field leakage.

### P28A-T3 Frontend Save Review Snapshot

Owner recommendation: Claude A or Codex F

Add a save action to the reviewed Trade Review / Agent Console surfaces after
the backend contract passes Codex B review.

Acceptance:

- Calls only approved report-save endpoint(s).
- Does not send raw broker/account/position data.
- Shows saved/failed/pending states without implying advice or execution.
- Reports page opens the saved artifact using existing Reports detail behavior.

### P28A-T4 Review And Closeout

Owner recommendation: Codex B for contract/privacy; Claude B or Codex F for
visual review if the UI changes materially; Claude E for Agent Team saved-output
boundary if role outputs are included.

Acceptance:

- Backend, frontend, and docs all agree that saved artifacts are immutable
  historical snapshots.
- Phase 28A does not activate interactive composer, streaming, broker actions,
  provider calls from frontend, or TradingAgents execution.

## Deferred

- Account-group CRUD and scope preference management.
- Report sharing/export.
- Regeneration/version comparison.
- Prompt/trace persistence.
- Tax-lot or transaction-level journal.
- Advice/execution workflow.

## Rollback

If the save path creates ambiguity, keep existing read-only report list/detail
behavior and remove only the new save endpoint/action. Existing Phase 27C saved
scope display remains valid for legacy/null reports.
